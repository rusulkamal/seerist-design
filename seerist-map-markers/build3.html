Build Approach

A phased plan for engineering to explore what's technically feasible, one layer at a time.

One layer at a time
Each phase adds exactly one visual dimension. If a layer causes problems, it can be held back without blocking the others.
Data-driven scope
Each phase produces real performance and rendering data that tells us whether the next layer is worth the added complexity before we commit to building it.
EOS-ready exit points
Any phase can be the shipping point. If the release timeline tightens, we stop at the last validated phase rather than shipping something half-finished.

Phase 1

Content Type & Breaking Pulse

Establish all four marker shapes and the one status signal (breaking pulse) in a single phase. Shape differentiates content type; the double-ring animation is the sole indicator for breaking events — no border changes or badges.

Alternative: Match the colours already used on content cards instead of a flat neutral — this would align map markers with the rest of the UI from day one rather than introducing colour in a later phase.

Build
  • Render all marker shapes on the map in a single neutral slate colour
  • Filled circle (30 px, border-radius: 50%) for events — shape signals approximate location
  • Rounded square with plain solid border for News & Social
  • Flat-top hexagon (clip-path polygon), blue #1D4ED8 outline, white fill for Analysis
  • Teardrop pin (border-radius: 50% 50% 50% 0, rotated −45°) for Assets — shape signals precise location
  • Cluster bubble for grouped events
  • Detect breaking status from event data
  • Apply double-ring CSS animation to breaking event markers only — outer ring 2.6×, inner 2.2×, opacity fades to 0 over 2s
  • Stagger pulse across markers (0.28s offset per marker); inner ring offset 0.5s within each marker
  • No border change, no badge — all event statuses share the same solid border
