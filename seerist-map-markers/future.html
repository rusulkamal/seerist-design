Future Ideas

Design decisions and behaviour improvements that are out of scope for the current build but worth capturing so they don't get lost.

Clustering & Visibility
Breaking events should never cluster
UX
Breaking events carry a pulsing ring that signals urgency. When they collapse into a cluster bubble, that signal disappears — the user loses visibility of something happening right now.
Rationale: A cluster count of "7" doesn't tell you one of those is breaking. Breaking events should always surface individually on the map regardless of zoom level, even if the surrounding events cluster around them.
Interactions
Clicking a cluster should show a mini-list before zooming
UX
Right now clusters zoom in on click. A first click could instead show a small popover listing the events inside (title + severity), letting the user decide whether to zoom or navigate directly to one.
Rationale: Avoids disorienting map jumps when the user only cares about one event inside a cluster of many.
Filter by severity should affect clustering radius
UX Data
When a user filters to high-severity events only, the reduced marker count makes aggressive clustering unnecessary. The clustering cell size could shrink to show more separation at the same zoom level.
Rationale: Clustering is a density problem. Filtering reduces density — the cluster threshold should adapt rather than stay fixed.
Marker Design
Stacked markers should show the highest severity on top
UX
When multiple events share the exact same coordinates, the current stack shows a count. The colour of the stack indicator should reflect the most severe event inside, not a neutral grey.
Rationale: If one of the stacked events is high-severity, the red signal should still surface at a glance — even before the user opens the stack.
